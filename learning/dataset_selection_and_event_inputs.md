# Dataset Selection And Event Inputs

This note explains the gap you noticed correctly:

the project can discover dataset pages, and the Lambda can ingest one trusted URL, but the repo does not yet define a canonical list of approved datasets to ingest.

That is an important architectural gap, not a misunderstanding on your side.

## The Current State

Repo fact: discovery is handled by [`exploring/datos.py`](/Users/rogermacedomilla/Documents/GitHub/peru-construcction-data-pipeline/exploring/datos.py).

Repo fact: ingestion is handled by the Lambda in [`app/lambda_app/handler.py`](/Users/rogermacedomilla/Documents/GitHub/peru-construcction-data-pipeline/app/lambda_app/handler.py).

Repo fact: a sample ingestion event lives in [`infra/events/html-dataset-page.json`](/Users/rogermacedomilla/Documents/GitHub/peru-construcction-data-pipeline/infra/events/html-dataset-page.json).

These are connected, but they are not the same thing.

## What `datos.py` Actually Gives You

`datos.py` discovers dataset pages from the public portal and prints entries such as:

- title
- dataset page URL
- page number
- organization
- topics
- formats

That means `datos.py` gives you candidate URLs and human-readable context.

It does **not** currently create:

- a permanent dataset registry file
- a stable `dataset_id`
- a stable `resource_id`
- a machine-approved list of datasets for ingestion

So today, `datos.py` is a discovery tool, not a configuration source.

## What The Lambda Event Needs

The Lambda event must contain:

- `dataset_id`
- `resource_id`
- `ingestion_id`
- `request`

You can see that in [`app/lambda_app/models.py`](/Users/rogermacedomilla/Documents/GitHub/peru-construcction-data-pipeline/app/lambda_app/models.py).

That means the Lambda expects someone upstream to already know:

- what this dataset should be called
- what resource within it is being fetched
- what URL should be fetched

General concept:

Discovery answers “what exists?”
Ingestion answers “what known thing should I fetch right now?”

The missing middle layer is often called:

- dataset registry
- approved source list
- ingestion catalog
- source manifest

## Why The Sample Event Can Feel Misleading

The sample event looks like the repo already knows the datasets:

```json
{
  "dataset_id": "licencias-construccion-lima",
  "resource_id": "dataset-page-html",
  "request": {
    "url": "https://www.datosabiertos.gob.pe/dataset/licencias-de-construccion"
  }
}
```

But that file is only one manually authored example.

It is not a central source of truth for all datasets.

So your intuition is correct: the repo does **not** yet define the official list of datasets to download.

## Do We Need To Define That?

Yes, if you want this project to become repeatable and operational.

Right now, the workflow is:

1. use discovery to find candidate dataset pages
2. manually choose one
3. manually create or edit an event payload
4. run ingestion

That is fine for the MVP and for learning.

But if you want repeatable ingestion in the cloud, you eventually need a defined source list.

## What That Future Definition Would Usually Contain

A simple dataset registry would likely include:

- stable dataset name
- stable resource name
- request URL
- optional notes about format
- optional source page URL
- optional organization
- whether the dataset is approved for ingestion

Example shape:

```json
{
  "source_id": "peru-open-data",
  "datasets": [
    {
      "dataset_id": "licencias-construccion-lima",
      "resource_id": "dataset-page-html",
      "url": "https://www.datosabiertos.gob.pe/dataset/licencias-de-construccion",
      "source_page": "https://www.datosabiertos.gob.pe/dataset",
      "approved": true
    }
  ]
}
```

That file does not exist yet in this repo.

## How To Work With The Repo Today

Today, the manual process is:

### 1. Discover candidates

```bash
uv run python exploring/datos.py --query "licencias construccion lima" --limit 10
```

### 2. Pick one dataset page URL from the output

For example:

```text
https://www.datosabiertos.gob.pe/dataset/licencias-de-construccion
```

### 3. Create an ingestion event

Use or copy [`infra/events/html-dataset-page.json`](/Users/rogermacedomilla/Documents/GitHub/peru-construcction-data-pipeline/infra/events/html-dataset-page.json) and change:

- `dataset_id`
- `resource_id`
- `request.url`
- `metadata`

### 4. Run SAM local invoke with that event

```bash
sam local invoke RawIngestionFunction \
  --template-file infra/cloudformation/raw_ingestion_template.yaml \
  --env-vars infra/parameters/sam-local-env.json \
  --event path/to/your-event.json
```

## How To Think About `dataset_id`

Right now, `dataset_id` is not assigned by the portal and not generated by the Lambda.

It is your own stable identifier for the dataset you decided to ingest.

A good `dataset_id` should be:

- readable
- stable across reruns
- not tied to one exact run timestamp

Example:

- good: `licencias-construccion-lima`
- weak: `dataset-1`
- bad: `2026-04-04-run-a`

Why:

`dataset_id` is part of the storage path. If you change it casually, you create a different storage location and break continuity.

## What Is Missing Architecturally

The missing piece is:

- not more Lambda logic
- not more SAM logic
- a curated list of approved datasets/resources

That list should sit between discovery and ingestion.

Once that exists, the Lambda can stay simple. It still just consumes trusted inputs.

## Recommended Mental Model

Use this model:

- `datos.py`: "what seems to exist on the portal?"
- event JSON: "what one dataset/resource do I want to ingest now?"
- future dataset registry: "what datasets have we approved as part of the pipeline?"
